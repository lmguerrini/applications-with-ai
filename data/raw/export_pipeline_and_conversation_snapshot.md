# Export Pipeline and Conversation Snapshot

## When This Matters

Use this document when conversation export output looks inconsistent, when you need to understand why exports survive Streamlit reruns, or when you are deciding what must be stored in each turn record so Markdown, JSON, CSV, and PDF exports stay stable.

This project does not build exports from live widget state. It exports from serialized conversation data. That distinction matters because Streamlit reruns the script repeatedly, and widgets are not a reliable source of truth for multi-format export.

## Recommended Pattern

The export pipeline works from stored turn records, not from the current rendered chat.

The stable sequence is:

1. build a turn record with `build_turn_record(...)`
2. append it to `st.session_state["conversation_history"]`
3. serialize the full history with `build_conversation_snapshot(...)`
4. derive one export artifact from that snapshot
5. send the artifact to `st.download_button(...)`

`build_conversation_snapshot(...)` is the key stabilizer. It converts `conversation_history` into canonical JSON with:

- `ensure_ascii=False`
- compact separators
- `sort_keys=True`

That snapshot is then passed into `build_cached_export_data(...)`, which is decorated with `@st.cache_data`. Caching works here because the input is a deterministic serialized snapshot, not a mutable Python list that changes shape across reruns.

The supported export formats are:

- Markdown
- JSON
- CSV
- PDF

Each format is generated by a dedicated helper:

- `build_conversation_markdown(...)`
- `build_conversation_json(...)`
- `build_conversation_csv(...)`
- `build_conversation_pdf(...)`

## Common Failure Modes

- Export content does not match the visible conversation.
  This usually means the turn records are incomplete, not that the exporter is broken. The exporters trust `conversation_history`.

- A rerun seems to change export content unexpectedly.
  The export path should stay stable if `conversation_history` is stable. If the snapshot changes, the export should change too. That is correct behavior.

- One format includes more detail than another.
  That is by design. The formats project different subsets of the turn record. JSON is closer to the raw structured data, CSV is intentionally flatter, and PDF is a readable text export rather than a full structured dump.

- Official-docs or tool details seem to disappear in export.
  Check whether the relevant fields are present in the turn record first. The export helpers can only serialize what was stored.

- PDF export fails on punctuation or markdown formatting.
  The PDF path sanitizes text with `normalize_text_for_pdf(...)` and strips a small set of markdown artifacts with `clean_markdown_text_for_pdf(...)`.

## Implementation Notes

The turn record built by `build_turn_record(...)` contains:

- `query`
- `answer`
- `used_context`
- `sources`
- `tool_result`
- `official_docs_result`
- `usage`

Those fields matter because they are the durable export source of truth, even though each exporter chooses a subset:

- Markdown includes query, answer, response type, sources, and tool result lines
- JSON includes query, answer, response type, `used_context`, sources, tool result, and usage
- CSV flattens response type, sources, tool name, tool payload JSON, and usage columns
- PDF includes question, response type, answer, sources, tool result details, and usage details

The export selection path in the sidebar is:

1. choose one format from `EXPORT_FORMAT_OPTIONS`
2. call `get_export_artifact(conversation_history, export_format)`
3. derive file name and MIME type
4. feed the result into `st.download_button(...)`

This is why export stays consistent across reruns. The app is not reconstructing export content from the currently open expanders or the last rendered widgets. It rebuilds artifacts from serialized data that already lives in `conversation_history`.

`build_cached_export_data(...)` also keeps export generation cheap when the user repeatedly downloads the same conversation in the same format.
